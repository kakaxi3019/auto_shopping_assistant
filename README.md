## 快速开始

### 1. 安装与启动

1. 下载安装包或解压便携版
2. 运行"自动购物助手"
3. 首次启动会显示"正在初始化服务，请稍候..."，等几秒就好

### 2. 配置 AI 模型

1. 点左侧 **设置**
2. 在"模型设置"里选 AI 提供商（OpenAI 或 Anthropic）
3. 填上 API Key、Base URL、模型名称
4. 点"验证连接"确认能用

### 3. 登录淘宝账号

1. 点左侧 **平台账号管理**
2. 点淘宝卡片里的"登录"
3. 在弹出的窗口里扫码或输密码登录
4. 登录成功窗口自动关闭

### 4. 同步历史订单

1. 选个时间范围（首次建议"近三月"）
2. 点"同步订单"
3. 等进度条跑完，会显示同步了多少条订单

### 5. 开始购物

1. 回到 **智能购物** 页面
2. 输入购物需求，比如"再买一箱牛奶"
3. 按 Enter 提交
4. 确认匹配结果后点"确认购买"
5. 在任务列表里看执行进度

几个关键设计点：

- **分层架构**：平台适配器分为基础设施层（浏览器/窗口/Cookie管理）、服务层（订单/搜索/购物车/结算/支付等业务逻辑）、协调层（流程编排）
- **双引擎浏览器**：主力用 Electron BrowserWindow，搞不定就换 Playwright Chromium 兜底
- **Cookie 双向同步**：Electron Session、本地文件、Playwright Context 三者之间实时同步登录态，保证每个窗口都是登录状态
- **事件驱动更新**：后端通过 `webContents.send` 推送任务状态，前端实时响应
- **平台适配器**：通过 `PlatformAdapter` 接口抽象平台操作，加新平台不用改核心代码

## 功能模块

### 智能购物

核心功能。输入一句"再买一箱牛奶"，就能自动完成从解析到购买的全流程。

**操作流程**：

1. 在输入框里输入购物需求，比如"再买一箱牛奶"、"买2瓶洗发水"
2. AI 解析你的指令，自动匹配历史订单里的商品
3. 预览面板展示匹配结果，你可以编辑、切换候选、改数量
4. 选好支付模式，点"确认购买"
5. 自动执行购买，实时显示进度
6. 买完发系统通知

**输入框**：

- Enter 快捷提交，Shift+Enter 换行
- 底部有最近 5 条成功购买记录，点一下就能快速复购

**预览面板**：

- 每个商品显示匹配状态、匹配方式（智能匹配/精确匹配/模糊匹配）、上次价格
- 如果候选订单价格差得比较多（>30%），会自动展开候选列表提醒你确认
- 可以在多个候选订单之间切换
- 可以搜索历史订单替换当前匹配
- 底部显示预计总金额

**任务状态**：

| 状态 | 含义 |
|------|------|
| pending | 排队中 |
| running | 正在买 |
| partial | 买到一半，有商品需要你确认 |
| success | 全部搞定 |
| failed | 全部失败 |
| cancelled | 已取消 |

> pending、running、partial 都算"执行中"。partial 意味着有些商品需要你手动确认或选替代品。

**任务卡片**：

- 进度条显示当前步骤（查找订单 → 再买一单 → 选择规格 → 提交订单 → 完成支付）
- 运行中显示已执行时间和实时日志
- 失败的商品会自动搜索替代品
- 可以重试失败项、再来一单、删除
- 日志里的超链接可以打开交互窗口手动操作

### 历史订单

从电商平台同步过来的订单数据，是智能匹配和"再买一单"的基础。

**功能**：

- 按平台查看订单列表（目前支持淘宝）
- 关键词搜索
- **再买一单**：直接基于历史订单创建购买任务，不用走 AI 解析
- **标记下架/恢复上架**：标记下架的订单不会出现在搜索匹配里
- 管理模式：批量选择和删除
- 按平台清除全部订单
- 分页浏览（每页 20 条）

---

### 平台账号管理

管理电商平台的登录状态和订单同步。

**登录**：

- 点登录后弹出淘宝登录窗口，扫码或输密码都行
- 登录成功后 Cookie 自动保存到本地
- 会显示 Cookie 有效期

**订单同步**：

- 可选时间范围：全部、近一周、近一月、近三月、近半年、近一年
- 实时显示同步进度（5 步进度条）
- 同步完显示订单数量和上次同步时间
- 增量同步，不会覆盖已有数据

---

### 定时任务

适合定期复购的商品，比如每周买牛奶、每月囤纸巾。

**重复方式**：

| 方式 | 说明 |
|------|------|
| 单次 | 执行一次就自动停 |
| 每天 | 每天固定时间执行 |
| 每周 | 每周指定星期几执行 |
| 每月 | 每月指定日期执行 |

**管理**：创建、编辑、删除、启用、暂停都支持。运行器每 60 秒检查一次，到期了就自动执行。

### 设置

#### 模型设置

- AI 提供商：OpenAI 或 Anthropic
- API 配置：API Key、Base URL、模型名称
- 连接验证：测试 API 能不能用
- 模型列表：验证成功后可以拉取可用模型

#### 支付设置

- **支付模式**：

| 模式 | 说明 |
|------|------|
| 自动支付 | 金额低于上限就自动免密支付，超了就弹窗让你手动付 |
| 确认金额后支付 | 自动加购提交订单，但付款要你自己来 |
| 仅加购 | 只加购物车，不结算 |

- **免密支付**：可开关，开启后可设置金额上限（默认 200 元）和适用平台
- **免打扰模式**：任务完成时不弹通知（后端支持，前端设置入口待完善）

## 用户操作指南

### 首次使用配置

用之前需要做三件事：

#### 第一步：配置 AI 模型

AI 模型负责把你的购物指令翻译成结构化数据，是智能购物的核心。

1. 点左侧 **设置**
2. 在"模型设置"标签页里：
   - 选 AI 提供商（OpenAI 或 Anthropic）
   - 填 API Key
   - 填 Base URL（用代理或自建服务的话改这里）
   - 填模型名称（比如 `gpt-4o-mini`、`claude-sonnet-4-20250514`）
3. 点 **验证连接**，确认显示"连接成功"
4. 验证成功后可以点 **获取模型列表**，从下拉框选模型
5. 点 **保存设置**

#### 第二步：登录电商平台

1. 点左侧 **平台账号管理**
2. 点淘宝卡片里的 **登录**
3. 在弹出的窗口里扫码或输密码
4. 登录成功窗口自动关闭，卡片会显示"已登录"和 Cookie 有效期

#### 第三步：同步历史订单

历史订单是匹配商品的基础，不同步就没法用购物功能。

1. 选同步时间范围：
   - **全部**：同步所有历史订单（比较慢）
   - **近三月**（推荐首次用）：同步最近 3 个月
   - 其他：近一周/近一月/近半年/近一年
2. 点 **同步订单**
3. 看着进度条跑完（5 步：同步订单 → 访问页面 → 等待渲染 → 解析数据 → 保存订单）
4. 完成后显示订单总数
5. 以后随时可以再同步，增量更新不会覆盖旧数据

### 智能购物操作步骤

#### 方式一：自然语言输入（推荐）

1. 在 **智能购物** 页面的输入框里输入购物需求
   - 比如："再买一箱牛奶"、"买2瓶海飞丝洗发水"、"补货洗衣液和纸巾"
   - Enter 提交，Shift+Enter 换行
2. 等一下，显示"解析中..."
3. 解析完会出现 **预览面板**：
   - 每个商品显示匹配状态（✅ 已匹配 / ❌ 未匹配）
   - 显示匹配方式和上次价格
4. 可以编辑匹配结果（可选）：
   - 改数量、搜索替换、切换候选订单、删除不需要的
   - ⚠️ 标记的商品建议确认一下
5. 选支付模式：
   - 自动支付：自动完成支付（限额内免密）
   - 确认金额后支付：自动下单，付款你自己来
   - 仅加购：只加购物车
6. 点 **确认购买**
7. 在任务列表里看进度

#### 方式二：快速复购

1. 输入框下面有 **最近购买** 区域
2. 点之前成功的购物指令，自动填入
3. Enter 提交，后续同方式一

#### 方式三：任务卡片"再来一单"

1. 找到之前的任务卡片
2. 点 **再来一单**
3. 直接创建新任务开始执行（跳过预览）

### 历史订单再买一单

从历史订单直接复购，不用输指令。

1. 点左侧 **历史订单**
2. 点 **淘宝** 进订单列表
3. 找到要复购的订单，点 **再买一单**
4. 自动创建任务并跳转到购物页面
5. 在任务列表里看进度

**其他操作**：

- **搜索**：输入关键词搜索订单
- **标记下架**：点"标记下架"，这个订单以后不再参与匹配
- **恢复上架**：已下架的可以点"恢复上架"
- **管理模式**：批量选择和删除
- **删除**：单条或批量删除

### 创建定时任务

适合定期复购，比如每周买牛奶、每月买洗衣液。

1. 点左侧 **定时任务**
2. 点 **创建定时任务**
3. 填写：
   - 任务名称：比如"每周牛奶"
   - 购物需求：比如"买一箱蒙牛纯牛奶"
   - 重复方式：单次/每天/每周/每月
   - 执行时间
   - 每周几（每周模式）或每月几号（每月模式）
4. 点 **创建**
5. 创建后可以随时启用/暂停/编辑/删除

### 购买中哪些情况需要你手动操作

虽然大部分流程是自动的，但有些情况会弹出一个窗口让你手动操作。以下是"再买一单"流程中会弹窗的场景：

#### 1. 商品规格变了，自动选不上

**什么时候出现**：点击"再买一单"后弹出了商品详情页，但原来的 SKU 规格已经对不上了（比如商家换了颜色选项、新增了规格），自动选择失败。

**弹窗内容**：一个淘宝商品页面，标题提示"请手动选择商品规格"。

**你要做什么**：在弹出的窗口里手动选好规格，点购买，然后关掉窗口。

**怎么重新打开**：如果窗口关了，可以在任务卡片的进度日志里点"弹出的窗口"这个链接重新打开。

#### 2. 遇到身份验证或安全校验

**什么时候出现**：淘宝检测到异常操作，弹出了滑动验证、短信验证、身份确认等页面。

**弹窗内容**：淘宝的验证页面。

**你要做什么**：完成验证（滑一下、输个验证码等），验证通过后自动流程会继续。

#### 3. 登录过期

**什么时候出现**：Cookie 过期了，操作时被重定向到了登录页。

**弹窗内容**：淘宝登录页面。

**你要做什么**：重新扫码或输密码登录，登录后可以重试任务。

#### 4. 操作超时

**什么时候出现**：点"再买一单"后 60 秒内没有完成操作（可能是页面加载慢、网络问题等）。

**弹窗内容**：当前页面，标题提示"自动购买未能完成，请手动完成购买"。

**你要做什么**：在弹出的窗口里手动完成剩余操作，然后关掉窗口。

#### 5. 需要手动付款

**什么时候出现**：
- 支付模式是"确认金额后支付"：每次提交订单后都会弹
- 支付模式是"自动支付"但金额超过了上限：弹窗让你手动付

**弹窗内容**：结算/支付页面，标题提示"请确认订单信息并提交"或"金额超过自动支付上限，请完成付款"。

**你要做什么**：确认金额，完成付款，然后关掉窗口。关掉后需要在任务卡片里点"确认已付款"或"未付款"。

#### 6. 商品下架或缺货

**什么时候出现**：点"再买一单"后发现商品已下架、已售罄。

**不会弹窗，但会**：在任务卡片里显示错误信息，并自动搜索替代商品。你可以在候选列表里选择一个替代品购买，或者点"标记下架"排除这个订单，或者点"不买了"。

### 处理购买异常

#### 商品已下架

1. 任务卡片显示"已下架"错误
2. 自动搜索替代商品并显示候选列表
3. 你可以：
   - **选择购买**：打开替代商品页面买
   - **我买了这个**：确认已经买了替代品
   - **标记下架**：排除原匹配订单
   - **不买了**：放弃

#### 价格异常

> ⚠️ 价格保护功能尚未实现，以下为设计规划。

1. 当前价格比历史价格涨幅超过阈值（默认 10%），自动拦截
2. 商品进入待确认清单
3. 在确认面板里决定继续买还是放弃

#### 登录过期

1. 任务失败显示"登录已过期"
2. 去 **平台账号管理** 重新登录
3. 登录后在任务卡片里点 **重试**

#### 重试失败项

1. 展开任务卡片
2. 找到失败的商品项，点 **重试**
3. 重新执行这个商品的购买流程

## 购买流程详解

### 整体流程概览

```
用户输入购物指令
       │
       ▼
  AI 解析指令 ──────→ 返回结构化购物清单
       │
       ▼
  匹配历史订单 ──────→ LLM 返回多个匹配订单 + 置信度排序
       │
       ▼
  用户确认/编辑 ─────→ 预览面板展示匹配结果
       │
       ▼
  执行购买流程
       │
       ├── 加购（addToCart）
       │     ├── 方式1: "再买一单"（主路径）
       │     ├── 方式2: 直接打开商品页（备用）
       │     └── 方式3: Playwright 模式（兜底）
       │
       ├── 价格保护检查 ──→ 涨太多就拦截（规划中，尚未实现）
       │
       ├── 结算（checkout）
       │
       └── 支付（pay）
             ├── auto_pay: 免密支付 / 超限弹窗
             ├── checkout_only: 弹支付窗口
             └── cart_only: 跳过
       │
       ▼
  完成 / 失败处理
       │
       ├── 成功 → 通知
       └── 失败 → 搜替代品 → 待确认
```

### 指令解析与订单匹配

#### AI 解析

输入购物指令后，`LlmParser` 调用 LLM 把自然语言翻译成结构化数据。如果用户有历史订单，LLM 会同时返回所有匹配的历史订单和置信度：

```json
{
  "items": [
    {
      "name": "牛奶",
      "quantity": 1,
      "orderRef": 5,
      "platform": "taobao",
      "matchedOrders": [
        { "orderRef": 5, "confidence": 95 },
        { "orderRef": 25, "confidence": 85 },
        { "orderRef": 20, "confidence": 70 }
      ]
    }
  ]
}
```

- `orderRef`：置信度最高的匹配订单 ID，作为默认选中项
- `matchedOrders`：所有匹配的历史订单列表，按置信度从高到低排列，最多 10 个
- `confidence`：0-100 的置信度分数，90+ 为完全匹配，70-89 为高度相关，50-69 为可能相关

解析器根据有没有历史订单用不同的提示词：

- **没有历史订单**：只提取商品名、数量、规格
- **有历史订单**：从数据库中取最近 100 条历史订单塞进提示词（不会把所有订单都发给 LLM，避免 token 消耗过大），让 LLM 找出所有相关订单并给出置信度排序。如果用户的历史订单超过 100 条，较早的订单可能不会被 LLM 看到，这时会通过数据库搜索来补充匹配。

#### 订单匹配策略

LLM 解析返回 `matchedOrders` 后，`previewCandidateOrders` 直接使用这些结果构建候选列表：

1. **LLM 匹配结果**（主路径）：直接使用 LLM 返回的 `matchedOrders`，按置信度排序，置信度最高的作为默认选中项
2. **数据库搜索补充**（兜底）：如果 LLM 没有返回 `matchedOrders`，则回退到数据库搜索（精确搜索 → 模糊搜索）

匹配结果会算歧义等级：

- **none**：只有一个候选订单，无歧义
- **low**：候选订单基本一致
- **high**：候选订单价格差 >30% 或商品名明显不同

高歧义的商品在预览面板会自动展开候选列表，提醒你确认。低歧义的默认折叠，你可以点开切换。

### 加购方式详解

三种加购方式，按优先级依次尝试：

#### 方式 1：再买一单（主路径）

用 Electron BrowserWindow 在隐藏窗口里操作，最可靠。

```
1. 拼订单详情页 URL
   https://trade.tmall.com/detail/orderDetail.htm?bizOrderId={orderId}

2. 创建隐藏窗口加载页面

3. 检查登录状态
   ├── 被重定向到登录页 → 重新同步 Cookie 重试一次
   └── 正常 → 继续

4. 看页面状态
   ├── 直接跳到结算页 → directToPay: true
   ├── 跳到购物车 → directToPay: false
   └── 历史订单归档页 → 失败（没有再买一单入口）

5. 找"再买一单"按钮
   ├── 在主文档和所有 iframe 里搜
   ├── 匹配文字：再买一单、再次购买、再来一单
   └── 同时用 CSS 选择器匹配

6. 检查是否下架
   ├── 页面有"已下架"/"宝贝不存在"等字样
   └── 且没有购买按钮 → 返回 out_of_stock

7. 点"再买一单"

8. 处理弹窗/跳转
   ├── 弹出商品详情页（有 SKU 参数）
   │   ├── 自动选 SKU 规格
   │   ├── 点"立即购买"或"领券购买"（优先）
   │   ├── 点"加入购物车"（次选）
   │   └── SKU 选不上 → 弹窗让你手动选
   ├── 弹出结算页 → directToPay: true
   ├── 弹出购物车 → directToPay: false
   ├── 弹出身份验证页 → 弹窗让你验证
   └── 弹出登录页 → 弹窗让你登录

9. 60 秒超时
   └── 超时就弹交互窗口让你手动完成
```

**SKU 自动选择**：

- 从 URL 参数里提取 `sku_properties` 和 `skuId`
- 在页面上找对应的规格按钮点一下
- URL 里没有 SKU 参数的话，尝试从页面上提取
- 选完点购买按钮（优先"立即购买"，排除"万人加购"之类的干扰按钮）

#### 方式 2：直接打开商品页（备用）

"再买一单"不行时，直接打开商品详情页加购。

```
1. 创建隐藏窗口加载商品 URL

2. 检查是否下架

3. 找"加入购物车"按钮
   ├── 有 SKU → 自动选规格后点
   └── 没 SKU → 直接点

4. 看跳转
   ├── 跳到购物车 → 加购成功
   ├── 跳到结算页 → directToPay: true
   └── 弹出 SKU 选择框 → 弹窗让你手动选
```

#### 方式 3：Playwright 模式（兜底）

前两种都不行时，用 Playwright headless Chromium 最后试一把。

```
1. 启动 Playwright Chromium
2. 加载 Cookie
3. 打开订单详情页
4. 找"再买一单"按钮（支持滚动重试）
5. 检测弹窗和跳转
6. 还是不行 → 弹手动操作窗口
```

#### 交互窗口

自动化搞不定时，会弹出一个 Electron 子窗口让你手动操作：

- 1100x800 大小，和主窗口共享 Cookie
- User-Agent 和主窗口一样
- 标题会提示你要做什么（比如"请手动选择商品规格"）
- 任务卡片的进度日志里有超链接可以重新打开

### 结算与支付

#### 结算流程

```
加购完成后
       │
       ├── directToPay: true（已经在结算页）
       │     └── 直接提交订单
       │
       └── directToPay: false（在购物车）
             ├── 点"结算"
             ├── 等确认订单页加载（最多轮询 15 次）
             ├── 等"提交订单"/"免密支付"等按钮出现
             └── 需要改数量的话 → 输入框设值或点加号
```

#### 三种支付模式详细流程

##### 自动支付（auto_pay）— 默认

尽量自动完成，金额超限才需要你动手。

```
1. 加购
   ├── 打开订单详情页 → 点"再买一单"
   ├── 自动选 SKU → 点"立即购买"
   └── 返回结果（当前价格、是否直接到结算页）

2. 价格保护（规划中，尚未实现）
   ├── 涨幅没超阈值 → 继续
   └── 涨幅超了 → 拦截，进待确认清单，结束

3. 结算
   ├── 已经在结算页 → 直接提交
   └── 在购物车 → 点"结算" → 等确认订单页

4. 算总金额 = 单价 × 数量
   │
   ├── 金额 ≤ 上限（默认200元）
   │     └── 自动点"免密支付"
   │         ├── 成功 → 搞定 ✅
   │         └── 失败 → 标记待付款，等你确认
   │
   └── 金额 > 上限
         └── 弹支付窗口
             ├── 你手动付款
             ├── 关窗后检测是否付成功
             ├── 付了 → 搞定 ✅
             └── 没付 → 标记待付款 ⏳

5. 待付款确认（如果需要）
   ├── 点"确认已付款" → 成功 ✅
   └── 点"未付款" → 失败 ❌
```

**前提**：自动支付依赖支付宝的"免密支付"功能，你得先在支付宝里开通小额免密，不然自动支付不了。

##### 确认金额后支付（checkout_only）

自动加购和下单，但付款你来。适合想先看看金额再付的场景。

```
1. 加购（同上）

2. 价格保护（规划中，尚未实现，同上）

3. 结算 → 提交订单

4. 弹支付窗口
   ├── 标题："请确认订单信息并提交"
   ├── 你在窗口里：
   │     ├── 看看金额对不对
   │     ├── 选支付方式
   │     └── 完成付款
   ├── 关窗后检测结果
   ├── 付了 → 搞定 ✅
   └── 没付 → 标记待付款 ⏳

5. 待付款确认（如果需要）
```

##### 仅加购（cart_only）

只加购物车，不结算不付款。适合先攒着等促销再一起结。

```
1. 加购
   ├── 打开订单详情页 → 点"再买一单"
   ├── 自动选 SKU → 点"加入购物车"
   └── 返回结果

2. 看结果
   ├── 成功加入购物车 → 搞定 ✅
   └── 直接跳到结算页（说明不支持加购物车）→ 失败 ❌

注意：仅加购模式不做价格保护检查，也不结算和支付
```

#### 三种模式对比

| 对比项 | 自动支付 | 确认金额后支付 | 仅加购 |
|--------|---------|--------------|--------|
| 加购 | ✅ 自动 | ✅ 自动 | ✅ 自动 |
| 价格保护 | 🔜 规划中 | 🔜 规划中 | ❌ 没有 |
| 结算 | ✅ 自动 | ✅ 自动 | ❌ 不做 |
| 支付 | ✅ 限额内自动 / 超限手动 | 🔧 每次手动 | ❌ 不做 |
| 需要你动手 | 仅超限时 | 每次要付款 | 不用 |
| 要不要开免密 | 要 | 不要 | 不要 |
| 适合 | 日常小额复购 | 想先看金额 | 攒着一起买 |

#### 待付款确认

需要你手动付款的商品会标记为 `pendingPayment`，在任务卡片里显示两个按钮：

- **确认已付款**：标记为成功
- **未付款**：标记为失败

### 异常处理与替代商品

#### 错误分类

| 错误类型 | 触发关键词 | 怎么处理 |
|----------|-----------|---------|
| `out_of_stock` | 已下架/已售罄/缺货/无法购买 | 搜替代品 |
| `login_expired` | 登录已过期/未登录 | 重新登录 |
| `not_supported` | 不支持再买一单/没找到入口 | 搜替代品 |
| `network_error` | 超时/网络错误/ERR_ | 可以重试 |
| `no_history` | 没找到历史订单 | 搜替代品 |
| `other` | 其他未分类错误 | 视情况处理 |

#### 替代商品搜索

商品缺货或不支持再买一单时，会自动搜替代品：

```
1. 提取搜索关键词
   ├── 去掉营销词（官方、旗舰、正品等）
   ├── 提取核心商品名
   └── 组合店铺名 + SKU 规格

2. 多策略搜索（找到 3-5 个就停）
   ├── "商品名 + 店铺名"
   ├── "核心商品名"
   ├── "商品名 + SKU规格"
   └── "精简关键词"

3. 存入待确认清单

4. 你来选择
   ├── "选择购买" → 打开商品页面买
   ├── "我买了这个" → 确认已买
   ├── "已自行解决" → 标记解决
   ├── "不买了" → 放弃
   └── "标记下架" → 排除原订单
```

### 价格保护机制

> ⚠️ 此功能尚未实现，以下为设计规划。

加购成功后、结算之前，会比较当前价格和历史价格：

```
当前价格 vs 历史价格
    │
    ├── 涨幅 ≤ 阈值（默认 10%）
    │     └── 正常买
    │
    └── 涨幅 > 阈值
          └── 拦截，进待确认清单
              ├── 提醒你价格异常
              └── 你来决定买不买
```

阈值在设置页面可以改，设成 0 就是关掉价格保护。

### 全自动购买的限制

这个工具的目标是尽量自动化，但说实话做不到 100% 无人值守。以下是主要的限制：

#### 1. 支付环节经常需要人

| 限制 | 原因 | 影响 |
|------|------|------|
| 免密支付有金额上限 | 支付宝小额免密有额度（默认 200 元），超了要输密码 | 大额订单自动不了 |
| 免密支付要提前开 | 你得在支付宝里主动开通"小额免密支付" | 没开就自动支付失败 |
| 有些银行卡不支持免密 | 某些银行卡/信用卡不支持免密协议 | 开了免密也可能要密码 |
| 人脸/指纹验证 | 支付宝有时会要求生物识别 | 自动化过不了 |

**建议**：选"确认金额后支付"模式，加购和下单自动来，付款你自己点一下。

#### 2. 商品规格可能对不上

| 限制 | 原因 | 影响 |
|------|------|------|
| 规格变了 | 商家改了 SKU 选项（换了颜色、尺寸等） | 原来的 SKU 参数匹配不上 |
| 新增了规格 | 商家加了原来没有的选项 | 缺参数，买不了 |
| SKU ID 失效 | 平台内部编号改了 | URL 里的 skuId 没用了 |

**怎么办**：SKU 选不上时会自动弹窗，你手动选一下就行。

#### 3. 淘宝的安全验证

| 限制 | 原因 | 影响 |
|------|------|------|
| 滑动验证码 | 淘宝觉得操作异常就弹验证 | 自动化过不了 |
| 短信验证 | 高风险操作要手机验证码 | 得你输 |
| 身份验证 | 有些订单要确认身份 | 得你点 |
| 风控拦截 | 操作太频繁触发反爬 | 账号可能被临时限制 |

**怎么办**：检测到验证页面会自动弹窗，你完成验证后自动流程继续。

#### 4. 登录会过期

| 限制 | 原因 | 影响 |
|------|------|------|
| Cookie 过期 | 淘宝登录态一般管几天到几周 | 过了就没法操作 |
| 异地登录检测 | 淘宝发现登录环境变了 | 要重新登录 |
| 多设备冲突 | 别的设备登录了同一账号 | Cookie 可能失效 |

**怎么办**：检测到登录过期会提示你重新登录，登录后可以重试。

#### 5. 商品本身的问题

| 限制 | 原因 | 影响 |
|------|------|------|
| 下架了 | 商家不卖了 | 买不了 |
| 没货了 | 暂时缺货 | 买不了 |
| 价格变了 | 促销结束或涨价 | 可能多花钱 |
| 页面改版 | 淘宝/天猫更新了页面结构 | 选择器可能失效 |

**怎么办**：
- 下架/缺货：自动搜替代品让你选
- 价格变了：价格保护功能规划中，目前不会自动拦截
- 页面改版：用了多种选择器策略和模糊匹配来提高容错

#### 6. "再买一单"本身的限制

| 限制 | 原因 | 影响 |
|------|------|------|
| 有些商品不支持 | 虚拟商品、定制商品等 | 主路径走不通 |
| 订单归档了 | 太老的订单进了归档 | 归档页没有"再买一单"按钮 |
| 限时商品 | 秒杀、预售等 | 不能常规复购 |

**怎么办**：会自动回退到直接打开商品页或 Playwright 模式，都不行就弹交互窗口。

#### 自动化程度总结

| 环节 | 自动化程度 | 说明 |
|------|-----------|------|
| 指令解析 | ✅ 全自动 | AI 解析自然语言 |
| 订单匹配 | ✅ 全自动 | 三级匹配自动查找 |
| 加购 | ⚠️ 大部分自动 | SKU 变了要手动选 |
| 结算 | ✅ 全自动 | 自动提交订单 |
| 支付 | ⚠️ 部分自动 | 小额免密自动，大额要手动 |
| 异常处理 | ⚠️ 半自动 | 自动检测，有些要你决定 |

> 日常小额复购（日用品、食品）选"自动支付"基本不用管；金额大或想确认的选"确认金额后支付"更稳妥。

## 开发与构建

### 环境要求

- Node.js >= 18
- npm >= 9

### 开发模式

```bash
npm install
npm run dev
```

### 生产构建

```bash
# 构建 ZIP 便携版
npm run build

# 构建安装目录（不打包）
npm run build:dir

npm run build:nsis
```

构建产物在 `release-dist/` 目录。

首次运行前需要配置，详见 [首次使用配置](#首次使用配置)。简单来说：

1. **AI 模型**：设置 → 模型设置 → 选提供商填 API Key
2. **平台账号**：平台账号管理 → 登录淘宝 → 同步历史订单
3. **支付模式**：设置 → 支付设置 → 选模式和上限
